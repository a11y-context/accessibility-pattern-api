## Purpose
A11y Context Project improves the accessibility of code generated by AI coding assistants by forcing the assistant to retrieve and apply proven accessibility best practices (patterns and global rules) before it writes code.

## What this system does
- **Enforcement:** Adds instructions that require the AI assistant to retrieve accessibility guidance before generating or editing code.
- **Retrieval:** Provides two ways to fetch guidance:
  - **MCP retrieval** (structured, deterministic tool calls)
  - **Docs retrieval** (your organization's indexing/RAG pipeline, or other doc-retrieval mechanism)
- **Stacks:** Patterns are organized by stack (for example web/react, android/compose). Retrieval should stay within the active stack to avoid cross-stack contamination.

## What you'll need
- Access to the repository you want to enable (write access to commit a small set of config/instruction files).
- An AI client you want to use (Cursor, Claude Code, Copilot, Android Studio Gemini, Gemini CLI, and so on).
- A decision on:
  - **Enforcement scope:** repo-scoped kit vs admin-managed (if supported in your environment)
  - **Retrieval mode:** MCP-only, docs-only, or MCP-first with docs fallback (recommended when you want resilience)

## Recommended default
Most teams should start with:
- **Repo-scoped kit** (portable, predictable, works across tools)
- **MCP-first with docs fallback** if your organization already indexes docs or plans to.
  - If you are not indexing docs yet, start MCP-only and add docs fallback later.

## How to use the rest of the installation docs
- **Installation:** The primary end-to-end setup flow, including enforcement choices, retrieval choices, installing the kit, and verification.
- **Enable in your client:** Tool-specific steps to ensure your AI client actually loads the repo instructions and can use MCP (if enabled).
- **Indexing guidance:** How to index this documentation (all stacks or a single stack) and tune chunking/retrieval so the AI gets clean, usable sections.
